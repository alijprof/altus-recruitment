<objective>
Ship PECR-compliant one-click unsubscribe for email campaigns and close the IN-02
personalisation-persistence gap. This is the PRE-LAUNCH BLOCKER flagged in
send-email-campaign.ts: the current unsubscribe link is a `mailto:` placeholder that
nothing processes into consent withdrawal, and Gmail/Yahoo bulk-sender rules require a
real one-click `List-Unsubscribe-Post` header.

Purpose: a candidate can withdraw marketing consent in one click, that withdrawal is
durable, and no future campaign re-emails them. Separately, the per-recipient AI
intro/outro that the send loop already generates gets persisted (columns already exist,
are never written).

Output: append-only migration (token + suppression columns), public no-auth unsubscribe
route (a SINGLE route.ts: GET serves an inline-HTML confirm page, POST suppresses — same
URL handles both the confirm form submit and the RFC 8058 one-click), real
List-Unsubscribe headers on every campaign email (with the header URL byte-for-byte equal
to the POST endpoint URL), suppression enforced in the segment query AND re-checked per
recipient at send time, and personalised_intro/outro persisted.
</objective>

## Trust Boundaries

| candidate browser / mail client → /unsubscribe/{token} | Fully unauthenticated public route; only the token is trusted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-0f4-BRUTE | Spoofing | token guessing on /unsubscribe | accept | Token is node:crypto randomBytes(>=32) base64url — ~256 bits entropy; brute-force infeasible. Rate-limiting deliberately skipped per locked decision; worst case an attacker who guesses a token only unsubscribes one candidate (no data read, no escalation). |
| T-0f4-ENUM | Information disclosure | invalid-token response | mitigate | Invalid token renders the SAME 200 inline-HTML page + constant copy ("This link is no longer valid"); POST always returns the same 200 confirmation HTML. No 404-vs-200 / timing / copy oracle reveals token validity. |
| T-0f4-PII | Information disclosure | confirm page / logs | mitigate | Confirm page shows only maskEmail(email) (escaped); no org name, no candidate name, no full email. Sentry tags are fixed strings only (layer/helper/subop) — never email or token (mirrors apply-form PII discipline). |
| T-0f4-PREFETCH | Tampering | mail-client link scanner auto-suppressing | mitigate | Suppression happens ONLY on POST, never on GET. Gmail/Outlook link pre-fetch is a GET and is side-effect-free; the recipient must click Unsubscribe (form POST to the same URL) or use the one-click POST header. |
| T-0f4-REDIRECT | Tampering | buildUnsubscribeUrl / route | mitigate | buildUnsubscribeUrl encodeURIComponent's the token path segment (no path-escape / open-redirect); POST returns a fixed in-app confirmation HTML, never redirects to a client-supplied URL. |
| T-0f4-XTENANT | Information disclosure / Elevation | token lookup → candidate write | mitigate | suppressByToken reads the recipient row by token, then scopes the candidate update by that recipient's own organization_id + candidate_id. No client-supplied org/candidate id is ever trusted (mirrors apply-form tenant-boundary rule). |
| T-0f4-IDEMPOTENT | Tampering | repeat unsubscribe | mitigate | suppressByToken re-reads current email_marketing_unsubscribed_at and only writes when NULL — repeat clicks / one-click + form double-fire do not overwrite the original withdrawal timestamp (audit integrity). |
| T-0f4-TOKENGAP | Tampering | legacy recipient with no token | mitigate | Send loop persists a generated token (error-checked) BEFORE building the URL; on persist failure the recipient is marked 'token_persist_error' and skipped — a campaign email is NEVER sent with a missing/empty unsubscribe token (which would ship a broken/PECR-non-compliant unsubscribe URL). |
| T-0f4-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages — node:crypto is stdlib. No package-install task in this plan; slopcheck not required. |

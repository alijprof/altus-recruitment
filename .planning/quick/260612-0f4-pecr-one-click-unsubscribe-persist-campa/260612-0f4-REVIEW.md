# Phase 260612-0f4: Code Review Report

**Reviewed:** 2026-06-12T10:00:00Z
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Reviewed the PECR one-click unsubscribe implementation across 8 files: migration, middleware public-path registration, unsubscribe helper library, unit tests, the public route handler, campaign DB helpers, the Inngest send loop, and the Resend email assembler.

The security model is sound in its fundamentals: service-role client for the public route, per-recipient 256-bit tokens, no status-code oracle for token validity, GET-has-no-side-effects, RFC 8058 `List-Unsubscribe-Post` header included, belt-and-braces suppression checks in both `getCampaignSegment` and the send loop, HTML escaping on all candidate-derived values in both the route and email assembler. These are all correct.

Three bugs remain. One is a user-facing correctness bug (wrong page shown for malformed POST with no token). One is a defence-in-depth gap in the token persist update (missing org-scope guard in the type narrowing). One is a data-exposure concern where `getCampaignWithRecipients` returns unsubscribe tokens to authenticated org members via the campaign detail page (tokens are not forwarded to the browser, but they transit the RSC render unnecessarily).

## Critical Issues

### CR-01: POST with empty/missing token returns success copy, not the "invalid link" page

**File:** `src/app/(public)/unsubscribe/[token]/route.ts:188-191`
**Issue:** The `GET` handler correctly returns `invalidTokenPage()` when `!token`. The `POST` handler at line 188 has the same guard but returns `confirmedPage()` — the success copy ("You have been unsubscribed") — instead. A user who somehow reaches `POST /unsubscribe/` without a token segment is told they have been unsubscribed when no suppression has occurred. More concretely: the RFC 8058 one-click POST is sent by the MUA to exactly the URL in the `List-Unsubscribe` header; that URL always contains the token. But any automation, scanner, or integration that strips the path segment and POSTs to the bare `/unsubscribe/` will receive a 200 with "You have been unsubscribed" — giving false confidence that suppression happened when it did not. This is a correctness bug (unsubscription silently failed) not a security issue.

**Fix:**
```typescript
// route.ts POST handler — replace:
if (!token) {
  return confirmedPage()    // BUG: sends success copy with no suppression
}

// with:
if (!token) {
  return invalidTokenPage()  // CORRECT: same as GET guard
}
```

## Warnings

### WR-01: Token persist update type narrowing prevents org-scope guard (defence-in-depth gap)

**File:** `src/lib/inngest/functions/send-email-campaign.ts:277-287`
**Issue:** The `sbTokenPersist` cast at line 277 only exposes a single `.eq()` method after `.update()`, resolving directly to `Promise<{ error }>`. This means the update on line 284–287 scopes only to `recipient.id`, with no second `.eq('organization_id', organization_id)`. Every other service-role update in this codebase adds both guards (e.g. `markCampaignFailed` at line 64, `step 'mark-sending'` at line 124). With service-role bypassing RLS, the UUID primary key makes collision impossible in practice — but the missing org guard breaks the established security pattern and cannot be caught by type-checking because the type actively prevents chaining a second `.eq()`. If this block is ever copy-pasted or the narrowing changes, the protection disappears silently.

**Fix:**
```typescript
// Extend the type narrowing to support two .eq() calls:
const sbTokenPersist = supabase as unknown as {
  from: (t: 'email_campaign_recipients') => {
    update: (p: Record<string, unknown>) => {
      eq: (c: string, v: string) => {
        eq: (c: string, v: string) => Promise<{ error: { message: string } | null }>
      }
    }
  }
}
const { error: persistErr } = await sbTokenPersist
  .from('email_campaign_recipients')
  .update({ unsubscribe_token: generatedToken })
  .eq('id', recipient.id)
  .eq('organization_id', organization_id)  // org-scope guard
```

### WR-02: `getCampaignWithRecipients` uses `select('*')` — unsubscribe tokens transit RSC render unnecessarily

**File:** `src/lib/db/campaigns.ts:256-259`
**Issue:** `getCampaignWithRecipients` selects all columns from `email_campaign_recipients` including `unsubscribe_token`. The campaign detail page (`src/app/(app)/campaigns/[id]/page.tsx:76`) calls this function with a session client. The resulting `CampaignWithRecipients` object (containing all tokens for all recipients) is held in the RSC render context. The `rows` mapping at line 96–104 of `page.tsx` correctly discards tokens — they never reach the browser. But the tokens transit the RSC data layer unnecessarily. Any future developer who expands `RecipientRow` or serialises `campaign.recipients` directly could inadvertently expose tokens to the browser. The `unsubscribe_token` column should only be selected by the service-role send loop, not by session-client reads.

**Fix:**
```typescript
// campaigns.ts — replace the recipients select:
const { data: recipients, error: recipientsErr } = await supabase
  .from('email_campaign_recipients')
  // Exclude unsubscribe_token — it is only needed by the service-role
  // send loop (send-email-campaign.ts), not by session-client reads.
  .select('id, campaign_id, organization_id, candidate_id, email, status, sent_at, error_message, resend_email_id, personalised_intro, personalised_outro, created_at, updated_at')
  .eq('campaign_id', campaignId)
  .order('created_at', { ascending: true })
```
Alternatively, add a separate `getCampaignWithRecipientsForSend` variant used by the Inngest function.

### WR-03: `assembleCampaignHtml` uses `encodeURI` instead of `escapeHtml` for the `href` attribute value

**File:** `src/lib/email/resend.ts:170, 185`
**Issue:** `safeUnsubUrl` is built with `encodeURI(unsubscribeUrl)` and placed directly in an HTML `href` attribute at line 185. `encodeURI` is a percent-encoder for valid URL characters — it does encode `<`, `>`, and `"`, but it does NOT encode `&`. If `NEXT_PUBLIC_SITE_URL` ever contained a `&` (e.g. query-string trickery), the `href` attribute value would contain a raw `&` which is invalid in HTML and is parsed as an HTML entity start by some email clients, potentially breaking the link. The correct approach is to HTML-escape the already-URI-encoded URL: `escapeHtml(encodeURI(unsubscribeUrl))`. Note: `NEXT_PUBLIC_SITE_URL` is validated as `z.string().url()` which prevents the worst cases, but the layering violation remains — `encodeURI` is a URL tool, not an HTML attribute tool.

**Fix:**
```typescript
// resend.ts line 170 — replace:
const safeUnsubUrl = encodeURI(unsubscribeUrl)

// with:
const safeUnsubUrl = escapeHtml(encodeURI(unsubscribeUrl))
// escapeHtml is defined above in this same file (line 116)
```

## Info

### IN-01: `suppressByToken` candidate re-read is redundant when the caller already has the current state

**File:** `src/lib/email/unsubscribe.ts:219-235`
**Issue:** `suppressByToken` performs an extra round-trip to re-read `email_marketing_unsubscribed_at` from the `candidates` table before deciding whether to write. This idempotency check is correct and intentional. However, the send loop at `send-email-campaign.ts:256` already checks `candidate.email_marketing_unsubscribed_at != null` and returns early before calling `suppressByToken`. So in the send-loop path the re-read inside `suppressByToken` is a redundant DB round-trip (step 2 in the helper). This is not a bug — `suppressByToken` is also called from the public POST handler where no pre-check exists and the idempotency re-read is necessary. This is a quality note: if `suppressByToken` is ever called from more callers, the double-read pattern is fine; if performance matters, a `skipIdempotencyCheck` option could be considered.

**Fix:** No action required. Note for future optimisation if DB call budget is a concern.

### IN-02: `getCampaignSegment` and the send loop suppress check use different null-check operators

**File:** `src/lib/db/campaigns.ts:70`, `src/lib/inngest/functions/send-email-campaign.ts:256`
**Issue:** The suppression re-check in `getCampaignSegment` uses PostgREST `.is('email_marketing_unsubscribed_at', null)` (filtering on IS NULL) while the send loop uses TypeScript `!= null` (i.e. `!== null && !== undefined`). Both are semantically correct — a non-null DB value will be a non-null string, and `undefined` never comes from a Supabase `maybeSingle()` data field. This is cosmetically inconsistent across the two check sites but causes no bug. Worth noting for readability when the types are regenerated after Task 4.

**Fix:** No action required.

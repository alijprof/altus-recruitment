## Every spec was opened and confirmed against current code

All 20 items were verified by reading the actual files (line numbers had drifted in places).
Two specs had **inaccurate root-cause diagnoses** (the symptom is real; the mechanism

### INACCURATE-1 ŌĆö Item 17 (LinkedIn ingest "queries run as anon role")
The route DOES authenticate the bearer via `supabase.auth.getUser(token)` (route.ts:111).
The real failure has TWO layers:
  (a) `/api/linkedin/ingest` is **NOT** in `PUBLIC_PATHS` (middleware.ts:8-34), so the
      Next.js middleware `updateSession` runs first, finds no Supabase cookie (request
      originates from `chrome-extension://`), `user === null`, `isPublic === false` ŌåÆ
      **307 redirect to /sign-in BEFORE the route handler ever executes.** This is the
      primary "always 500/dead" cause (the extension sees a redirect, not JSON).
  (b) Even past the middleware, `createClient()` builds a cookie-bound SSR client.
      `getUser(token)` validates the token but does NOT attach it to the client's
      PostgREST `Authorization` header. So `getProfile` + `upsertCandidateFromLinkedIn`
      run with no session ŌåÆ `auth.uid()` NULL ŌåÆ anon role ŌåÆ RLS blocks ŌåÆ 500.
**Correct fix (Task 7):** add the route to `PUBLIC_PATHS` AND build a token-scoped client
(global `Authorization: Bearer <token>` header) so queries run as that user under RLS.
This keeps RLS as the tenant guard ŌĆö the safest, most tenant-safe option. The existing
`upsertCandidateFromLinkedIn` already asserts the dedup row's org defensively.

### INACCURATE-2 ŌĆö Item 11 (first-time Outlook connect "UPDATE fallback no-ops")
The mechanism is subtler than stated. `upsertOutlookCredentials` does a plain INSERT
WITHOUT passing `organization_id` (outlook-credentials.ts:152-161), relying on the
`set_organization_id` trigger. Under service-role (no session) the trigger RAISES
(`organization_id is required and could not be resolved from auth context`), so the INSERT
fails. The callback then runs an UPDATE fallback (callback/route.ts:166-178) ŌĆö but on a
**first-time** connect there is no existing row, so the UPDATE matches 0 rows and
PostgREST returns `error: null` ŌåÆ code treats it as success ŌåÆ "connected" while nothing
persisted.
**Correct fix (Task 6):** add an optional `organizationId` param to
`upsertOutlookCredentials` and include it in the insert payload (the callback already
resolved `organizationId` at route.ts:146). This makes the INSERT succeed on first connect.
Also check the UPDATE fallback row count and treat 0 rows as `persist_failed`.

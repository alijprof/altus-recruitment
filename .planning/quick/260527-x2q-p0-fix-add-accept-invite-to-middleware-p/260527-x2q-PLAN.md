# Quick 260527-x2q — P0 fix: `/accept-invite` middleware bypass

**Discovered by:** Pre-UAT browser-automation pipeline against https://altus-recruitment.vercel.app
**Severity:** P0 — invitation flow completely broken for new users
**Fix:** 1 line. Add `/accept-invite` to `PUBLIC_PATHS` in `src/lib/supabase/middleware.ts`.

## The bug

Original 260524-bpy shipped `/accept-invite/[token]/route.ts` to handle invite-link clicks. The route:
1. Looks up the token via service-role
2. Sets `altus_invite_token` httpOnly cookie
3. Redirects to `/sign-in?email=...&invite=1`

But `src/lib/supabase/middleware.ts` `PUBLIC_PATHS` did NOT include `/accept-invite`. So for unauthenticated users (the common case — invitees don't have Supabase sessions yet):

1. User clicks invite email link → GET `/accept-invite/[token]`
2. Middleware: `pathname` not in PUBLIC_PATHS → redirect to `/sign-in?next=/accept-invite/[token]`
3. Route handler never executes → cookie never set
4. User completes magic-link sign-in
5. `/auth/callback` finds no `altus_invite_token` cookie → falls through to `handle_new_user` trigger → bootstraps fresh org
6. Invitee lands in their own brand-new org, NOT the inviter's

## Why all 4 code reviews + /security-review missed it

Each review focused on the route handler in isolation. None cross-checked the middleware allowlist. The browser-automation walk (per HARD RULE #1) caught it because it actually exercised the unauthenticated-user path against the deployed URL.

## Verification post-fix

After Vercel auto-deploys this commit, re-run `vercel:verification` against `/accept-invite/[token]`:
- Should redirect to `/sign-in?email=...` (with cookie set) when valid
- Should redirect to `/sign-in?error=invalid-invite` when token unknown/expired

## Out of scope (defer)

- WR-01 from 260524-iav review: `/accept-invite/[token]/route.ts:56` still appends a stale `&invite=1` to its redirect URL. Now dead code (the consumer was removed in 260524-iav B2). Cosmetic.
- The middleware `PUBLIC_PATHS.some(...)` uses `pathname.startsWith(\`${p}/\`)` so `/accept-invite/[any-token]` matches `/accept-invite/` prefix — no change to the matcher needed.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server action | Untrusted JSON crosses into `submitFeedbackAction`; body length / type must be validated server-side |
| server action → Resend API | Outbound HTTP to third-party; failure must not break user-visible flow |
| user-controlled string → external email | `body`, `page_url`, `user_agent` get embedded into an outbound email; must not be interpreted as HTML/markup |
| client browser → page_url / user_agent | Both are window-side strings the user could spoof; server treats them as untrusted text only (no parsing / no eval) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-260524-b6v-01 | Tampering | `submitFeedbackAction` body input | mitigate | Zod `safeParse` with `.trim().min(1).max(2000)`; DB CHECK constraint `length(body) between 1 and 2000` enforces a second layer |
| T-260524-b6v-02 | Information Disclosure | RLS on `public.feedback` | mitigate | SELECT policy `organization_id = public.current_organization_id()`; INSERT policy adds `submitted_by = auth.uid()` so users can't impersonate other org members |
| T-260524-b6v-03 | Spoofing | `submitted_by` / `organization_id` on insert | mitigate | Server action passes `submitted_by: user.id` from `auth.getUser()`; `organization_id` is NEVER sent from the client — `_set_org` trigger auto-fills from auth context; RLS WITH CHECK validates both |
| T-260524-b6v-04 | Information Disclosure | PII in Sentry | mitigate | Sentry captures only the error object + static `{ feature: 'feedback' }` tag — feedback body, user email, and user-agent string never enter Sentry payloads (CLAUDE.md rule) |
| T-260524-b6v-05 | Tampering | HTML injection via outbound email | mitigate | Send Resend `text:` (plaintext) only; do NOT populate `html:` with user-controlled strings. If a future iteration adds HTML, escape body / page_url / user_agent first |
| T-260524-b6v-06 | Denial of Service | Spam submissions from authenticated user | accept | Anchor is a 2-3 person agency — rate-limiting feedback is unnecessary at this scale. If abuse appears later, add Inngest-based throttle keyed on `(submitted_by, minute)` |
| T-260524-b6v-07 | Repudiation | Who submitted what | mitigate | `submitted_by` + `created_at` columns are non-null; insert is append-only (no UPDATE / DELETE RLS policies) |
| T-260524-b6v-08 | Denial of Service | Resend outage blocks user feedback | mitigate | `sendResendEmail` is wrapped in try/catch inside the server action; failure logs to Sentry and the action still returns `{ ok: true }` because the DB row is canonical |
| T-260524-b6v-09 | Information Disclosure | RESEND_API_KEY leakage | mitigate | Declared in `env.ts` `server` block (server-only — never bundled into client). `sendResendEmail` only runs from the server action context |
</threat_model>

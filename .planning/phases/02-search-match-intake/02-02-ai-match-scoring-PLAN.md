**Success criterion satisfied:** ROADMAP #2 — "Recruiter can open a job and see the top-matching candidates surfaced automatically, each with a 0–100 score, 2–3 strengths, 1–2 gaps, and suggested screening questions generated by Sonnet"
**Mode:** mvp — vertical slice (Inngest precompute fires on job create → cache lookup → Sonnet call → results render on `/jobs/[id]/matches` with full explanations within 60 s of opening a new job)

## Goal

After this plan, opening `/jobs/[id]/matches` shows the top 10 candidates each with: score (0-100, colour-coded badge), 2-3 strengths, 0-2 gaps, exactly 3 screening questions, and a confidence indicator. Score generation is hybrid: a background Inngest function precomputes scores for the top 10 vector-similar candidates as soon as a job is created or its embedding is refreshed; on page render, any cache miss is filled by an on-demand "Explain" action that streams in within ~5 s. Cache keys include both `candidate_embedding_version` and `job_embedding_version`, so any material change invalidates the relevant rows. A per-org monthly Sonnet spend ceiling (configurable, default £100) surfaces a Sentry warning when crossed.

## Phase Goal (MVP user story)

**As a** recruiter, **I want to** open any job and immediately see ranked candidates with score, strengths, gaps, and screening questions written by AI — **so that** I know who to call first without reading every CV.

**Implementation:**

1. **`precompute-matches-for-job.ts`** Inngest function:
   - `id: 'precompute-matches-for-job'`
   - `triggers: [{ event: 'job/score-top-candidates' }]`
   - `concurrency: { limit: 2, key: 'event.data.organization_id' }` (each step does ~10 Sonnet calls — tighter than embed concurrency)
   - `retries: 3`
   - Event payload: `{ organization_id: string; job_id: string; user_id: string | null }`
   - Event-shape guards FIRST (outside any step.run): both fields present → else `NonRetriableError('missing required fields')`. Then look up the job via service-role; assert `job.organization_id === event.data.organization_id`; else `NonRetriableError('job not in claimed organization')` (per Phase 1 LEARNINGS — service-role bypasses RLS so this check is the boundary).
   - Step 1: `read-job-context` — service-role select of: job's columns needed for `jobEmbeddingText` (or precomputed summary), `job.embedding_version`, current month's `ai_usage` total via `getOrgMatchSpendThisMonth`.
   - **Cost-ceiling check INSIDE step 1.** Read `env.MAX_MONTHLY_MATCH_SPEND_PENCE` (new optional env var declared in Plan 0; if not set, hard-coded fallback of 10000 = £100 per RESEARCH §B.8). If the org's month-to-date `match_score` spend exceeds the ceiling: `Sentry.captureMessage(\`match scoring spend ceiling reached for org \${orgId}\`, { level: 'warning', tags: { layer: 'inngest', function: 'precompute-matches-for-job', org_id: orgId } })` AND return early without scoring. Do NOT throw — that would retry. Cache stays as-is; recruiter still sees vector-only matches.
   - Step 2: `select-top-candidates` — `getTopCandidatesByVector(supabase, { jobId, limit: 10 })` (Plan 0 helper). If returns `[]` (job has no embedding yet, or no candidates have embeddings), return early with a Sentry breadcrumb noting "no candidates to score".
   - Step 3: For EACH candidate (loop, NOT parallelized — sequential within the step to bound concurrency): `step.run(\`score-\${candidateId}\`, async () => { ... })`. Inside each inner step:
     1. Cache lookup: `getMatchSummary(supabase, { candidateId, jobId, candidateEmbeddingVersion: <fetched>, jobEmbeddingVersion: <fetched> })`. If `.ok && .data` (hit), return early — no Sonnet call.
     2. Build inputs: `candidateSummary` = the candidate's structured fields + first 2k chars of latest CV text (D2-08 cap). `jobSummary` = job's structured fields + JD body capped at 4k chars.
     3. Call `scoreCandidateForJob({ candidateSummary, jobSummary, organizationId, userId })` — this writes `ai_usage` automatically via `runWithLogging`.
     4. Upsert via `upsertMatchSummary(supabase, { ... content: matchScore, model: 'claude-sonnet-4-6', costPence: <estimate from same calc as claude.ts — use 1p as a defensive default if estimating server-side is awkward> })`. The `cost_pence` on `ai_summaries` is bookkeeping for the matches page; the authoritative `ai_usage` row is already written by the wrapper.
   - Independent inner steps mean Inngest retries each candidate's score independently on transient Anthropic errors — no whole-job retry.
   - `onFailure` handler at the function level uses the PII-safe pattern from `parse-cv.ts:110-133`: extract `originalData`, `Sentry.captureException(new Error(\`\${error.name}: \${readStatus(error)}\`), { tags: { layer: 'inngest', function: 'precompute-matches-for-job', job_id: originalData.job_id } })`. Do NOT pass `error` directly.

2. **`embed-job-on-jd-change.ts` chain.** After Step 3 (persist embedding) successfully writes the row, append a Step 4: `step.sendEvent('rescore-after-embed', { name: 'job/score-top-candidates', data: { organization_id, job_id, user_id } })`. Inngest's `step.sendEvent` is the canonical way to chain events idempotently (it dedupes on retry). The cache-key in `ai_summaries` will see the new `job_embedding_version` and recompute only the affected rows.

3. **`clients/[id]/jobs/new/actions.ts`.** After the existing `inngest.send('job/embed', ...)`: do NOT send `job/score-top-candidates` directly here. The embed function chains it (step 2 above). Document this in a comment.
   - Rationale: chaining inside the Inngest function (instead of from the action) gives us per-attempt idempotency AND avoids the "embed isn't ready yet → score returns empty" race. The action only fires one event; Inngest handles the rest.

4. **`getOrgMatchSpendThisMonth(supabase, orgId)`** in `ai-summaries.ts`:
   - SELECT `coalesce(sum(cost_pence), 0)` FROM `ai_usage` WHERE `organization_id = $1 AND purpose = 'match_score' AND created_at >= date_trunc('month', now())`.
   - Returns `DbResult<number>` (pence).

5. **`readStatus` lift.** If `src/lib/observability/inngest.ts` doesn't exist yet from Plan 1, create it now with the `readStatus(err: unknown): number | 'unknown'` helper copied from `parse-cv.ts:89-99` plus a `formatErrorForSentry(err: unknown, prefix?: string): Error` helper that returns `new Error(\`\${name}: \${status}\`)` so callers can shorten the boilerplate. Update `parse-cv.ts` to import from the new module (small refactor; no behaviour change). `precompute-matches-for-job.ts` imports from there.

## Surface-by-Surface Interaction Contract

### Surface 1: Voice Note Button on `/candidates/[id]`

**Placement (Claude's Discretion — decided here):** Add a `Button` with `variant="outline"` labeled "Voice note" with a `Mic` icon, placed in the `CandidateDetailHeader` action row alongside existing "Log activity" and "Edit" buttons. This is the least-disruptive placement — it sits with other candidate-scoped actions at the top of the detail page. No floating action button (FAB) — the existing page has no FAB pattern and adding one conflicts with the mobile nav drawer.

**States:**
- **Idle:** `<Button variant="outline" size="sm"><Mic className="size-4" aria-hidden /> Voice note</Button>` — navigates to `/candidates/[id]/voice-notes/new`
- **Existing unreviewed notes:** Show a yellow dot badge on the button (`relative` + `absolute top-0 right-0 size-2 rounded-full bg-amber-500`) indicating pending review. This mirrors the existing pattern used for pending CV parse reviews.

**Voice note capture page (`/candidates/[id]/voice-notes/new`):**
- Reuse `MicRecorder` component verbatim from `src/app/(app)/spec/new/mic-recorder.tsx`. Change button label from "Record this spec call" to "Start voice note".
- Page heading: "Voice note — [Candidate full name]" at `text-2xl font-semibold`.
- File upload fallback label: "Or upload an audio file" — mirrors spec upload form pattern.
- Submit button: "Submit for processing" (`Button` default variant) — disabled until a file is selected or a recording is captured.
- Processing state after submit: inline `<p role="status" aria-live="polite" className="text-muted-foreground text-sm">Processing your note… this usually takes under 30 seconds.</p>` — matches `SendCheckinModal` polling copy pattern.

### Surface 2: Voice Note Review (`/candidates/[id]/voice-notes/[vnid]/review`)

**Layout:** Single-column card, `max-w-2xl mx-auto`, consistent with spec-review page layout.

**Per-field approval table:**
- Each proposed field change renders as a table row: `Checkbox` (default: checked) | Field name (`text-sm font-semibold`) | Before value (`text-muted-foreground text-sm` with strikethrough `line-through`) | Arrow `→` | After value (`text-sm font-semibold`).
- `notes` (append-only) renders as: Checkbox | "Append to notes" | (no before value) | Quoted append text in `text-sm font-normal`.
- `activity_kind` + `activity_body`: shown as a read-only summary card below the table (`bg-muted/40 rounded-md border p-3`) — not checkboxable, always applied if any changes approved.
- Action items list: read-only, `<ul className="list-disc pl-4 space-y-1 text-sm">` below the activity summary.

**CTA row (sticky on mobile, fixed at bottom of form):**
- Primary: `<Button>Apply N changes</Button>` — N = count of checked rows, updates reactively. Disabled when 0 checked.
- Secondary: `<Button variant="outline" className="text-destructive hover:bg-destructive/5">Reject all</Button>` — opens an `AlertDialog` confirmation before firing server action.
- The "Apply changes" label counts only field-change rows, not the activity row (activity always logs if any approval proceeds).

**Processing states:**
- While Inngest is still transcribing: show the `pending` / `transcribing` status panel (amber pulse dot + "Processing your voice note… Check back in a moment.").
- On `ready_for_review`: show the approval table.
- On `applied`: show `bg-green-50 rounded-md border border-green-200 p-4 text-green-800 text-sm` success banner "Changes applied successfully." with a "Back to candidate" link.
- On `failed`: show destructive alert with "Something went wrong processing this voice note. Please try again or log your note manually."

### Surface 3: Campaign Builder (`/campaigns/new`)

**Layout:** Three-step wizard. Use shadcn `Tabs` component with step labels "Segment", "Message", "Review & send". Steps are linear — "Message" tab disabled until segment has at least 1 recipient; "Review & send" tab disabled until subject + body are non-empty.

**Step 1 — Segment:**
- Campaign name: `Input` field, label "Campaign name", placeholder "Q3 actively-looking outreach".
- Segment filter: `Select` (multi-value pattern via `Command` + `Popover`) for `market_status`. Label "Send to candidates who are". Options map from existing `market_status` enum using `MarketStatusBadge` labels.
- GDPR note: `<p className="text-muted-foreground text-xs mt-2">Only candidates with active GDPR consent are included. Candidates who have withdrawn consent are automatically excluded.</p>`
- Recipient count preview: loads on filter change via server action — shows `<p className="text-sm font-semibold mt-3">{count} candidate{count !== 1 ? 's' : ''} match this segment</p>` or `<p className="text-muted-foreground text-sm mt-3">No candidates match this segment. Adjust the filters.</p>`
- "Continue to message" `Button` (default variant), disabled when count is 0.

**Step 2 — Message:**
- Subject: `Input` field, label "Email subject", `maxLength={200}`.
- Body template: `Textarea` with `rows={12}`, label "Email body". Helper text below: `<p className="text-muted-foreground text-xs">Write the main body. A personalised introduction and sign-off will be added per recipient by AI.</p>`
- Preview of personalisation: `<details>` element with summary "How personalisation works" + a short explanation. Not a live preview (live preview is per-recipient, expensive — deferred per D4-07 scope).

**Step 3 — Review & send:**
- Summary card: campaign name, segment description, recipient count, subject line preview.
- Recipient table: `Table` component, columns: Name | Market status (`MarketStatusBadge`) | Last active. Paginated at 20 rows with "Show more" button if > 20 recipients.
- MARKET-03 gate — explicit confirmation required: `<AlertDialog>` triggered by "Send campaign" button. Dialog title: "Send this campaign?". Body: "This will send {N} personalised emails via Resend. This action cannot be undone." Confirm button: "Send {N} emails". Cancel button: "Go back".
- After approval: inline progress indicator replaces the button: `<Progress className="h-2" value={sentPercent} />` + status text "{sent} of {total} sent" — polling the campaign status via a lightweight server action every 3 seconds.
- Per-recipient status column appears in the table once sending begins: `sent` (green tick icon), `failed` (red X icon), `pending` (grey dash), `failed_cap_exceeded` (amber warning icon with tooltip "AI usage cap reached").

### Surface 4: Campaigns List (`/campaigns`)

**Layout:** Standard list page, consistent with existing `/spec`, `/jobs`, `/clients` list patterns. No custom layout — use the `max-w-5xl mx-auto space-y-6` pattern from buyer-value page.

- Page heading: "Campaigns" at `text-2xl font-semibold`.
- "New campaign" button: top-right, `Button` default variant, `Plus` icon.
- Table columns: Name | Status (badge) | Recipients | Sent | Created | Actions (dropdown).
- Empty state: `EmptyState` component with heading "No campaigns yet" + body "Build a segmented email campaign to reach candidates at the right moment." + CTA `{ href: '/campaigns/new', label: 'Create campaign' }`.

### Surface 5: FollowUpWidget Enhancement (REMIND-01 quick-action CTA)

**Existing pattern:** `FollowUpWidget` in `src/app/(app)/_dashboard/follow-up-widget.tsx` — each row is a `Link` to `/candidates/[id]`.

**Enhancement (Claude's Discretion — decided here):** Add an inline quick-action button to each row without requiring navigation. Pattern mirrors the `DormantClientsWidget`'s "Send check-in" button approach.

**Row layout change:**
```
[Candidate name + days since contact] [MarketStatusBadge] [Log call button]
```

- "Log call" renders as `<Button variant="ghost" size="sm">` with a `Phone` icon + "Log call" label. On click, opens a lightweight `Dialog` (not a full navigation) with:
  - Dialog title: "Log call — [candidate name]"
  - `Textarea` with `rows={3}`, label "Call notes (optional)", placeholder "Brief summary of what you discussed."
  - `Button` default "Log call" — calls `logActivityAction` with `kind='call'`. On success: close dialog + show sonner toast "Call logged".
  - `Button` variant="ghost" "Cancel".
- The row link (name/days) still navigates to the full candidate detail page for users who want the full context.
- "Log call" button stops click propagation so tapping it doesn't also trigger the row link navigation.

**Empty state (unchanged):** existing "No follow-ups due" pattern retained.

### Surface 6: NL Reporting (`/reports/nl`)

**Layout:** Consistent with existing `/reports/buyer-value` page. Use `max-w-5xl mx-auto space-y-6`. Back link to `/reports` with `ChevronLeft` icon — same as buyer-value page.

- Page heading: "Ask a question" at `text-2xl font-semibold`.
- Subheading: `<p className="text-muted-foreground text-sm">Ask about your desk in plain English. Powered by a curated library of validated queries.</p>`

**Query input form:**
- `Textarea` with `rows={2}`, `placeholder="e.g. How many placements did we make last quarter by sector?"`, label "Your question".
- Submit button: `Button` default "Ask" with a `Search` icon. Loading state: "Asking…" with a `Loader2` spinning icon. Disabled while loading.
- Below the textarea: `<p className="text-muted-foreground text-xs">Questions are matched to pre-validated report templates — no free-form SQL.</p>` (transparency on D4-08 safety model)

**Result display:**
- Matched template transparency (D4-08 requirement): `<p className="text-muted-foreground text-xs">Matched template: <span className="font-medium text-foreground">{matchedTemplate}</span></p>` shown above the results table.
- Results: `Table` component with dynamic columns from the RPC response. Column headers from keys of first row, formatted as sentence case.
- Numeric columns: `tabular-nums text-right` alignment.
- Row count summary: `<p className="text-muted-foreground text-xs">{rows.length} rows returned</p>` below the table.

**No-match state:** `<div role="alert" className="rounded-md border bg-muted/40 p-4 text-sm">` — "No matching report template. Try rephrasing your question, or choose from the example questions below." + a list of 3 example questions as `<button className="text-sm underline">` that pre-fill the textarea.

**Error state:** `<div role="alert" className="border-destructive/30 bg-destructive/5 text-destructive rounded-md border p-3 text-sm">` — "Couldn't run that report. {error message}. Please try again."

**Reports nav entry:** Add "Natural language" as a card on the existing `/reports` page alongside "Buyer-value report" and "Source attribution".

## Phase 4 AI-State UI Patterns

These patterns govern how the UI communicates asynchronous AI processing state across all four slices. They must be consistent across surfaces.

### AI processing states (universal pattern)

All AI-backed async operations use this state machine for user-facing display:

| State | Visual treatment | ARIA |
|-------|-----------------|------|
| Not started | Normal idle UI | — |
| Requested / polling | `<p role="status" aria-live="polite" className="text-muted-foreground text-sm">` + descriptive copy | `role="status"` + `aria-live="polite"` |
| Timeout (>10s no response) | Same container + retry button | `role="alert"` on timeout message |
| Ready | Transition to content | — |
| Error | `role="alert"` destructive banner | `role="alert"` |

Source: `SendCheckinModal` pattern — use verbatim across voice note processing, campaign preview load, and NL query execution.

### AI cost transparency

For campaign personalisation specifically: show per-campaign estimated cost below the recipient count on the Review & send step:
`<p className="text-muted-foreground text-xs">Estimated AI cost: ~£{estimatedCostPounds.toFixed(2)} ({recipientCount} × ~£0.002 per recipient)</p>`

This is informational only — no blocking gate. The cap enforcement is handled server-side.

## Accessibility Requirements

These are minimum bars — not aspirational:

- All interactive elements have explicit `aria-label` when the visible label is absent (icon-only buttons, status dots).
- `role="alert"` on all error states that appear dynamically (voice note failed, campaign send error, NL query error).
- `role="status"` + `aria-live="polite"` on all AI processing states.
- `Checkbox` rows in the voice note review form must be keyboard-navigable: Tab to each row, Space to toggle. Use shadcn `Checkbox` component (Radix-backed, accessible by default).
- Campaign send `AlertDialog`: focus traps correctly — shadcn `AlertDialog` is Radix-backed and handles this.
- Progress bar during campaign send: `<Progress aria-label="Campaign send progress" value={sentPercent} />` with an accompanying text label for screen readers.
- `MarketStatusBadge` uses `border-transparent` to avoid color-only status communication — the text label is always present.
